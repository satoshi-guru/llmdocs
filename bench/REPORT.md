# Benchmark — token × quality

> **Provenance & reproduction.** The `fastapi`/`pydantic` rows below were generated by the author on
> a local store fetched with the current pipeline; they are **not** CI-gated. CI only re-runs the
> bundled `alpha-lib` fixture (`store/example`). To reproduce the large-library figures yourself:
> `python crawler.py --preset fastapi && python crawler.py --preset pydantic`, then
> `python bench/bench.py --libs fastapi pydantic --store ~/.llmdocs/docs --out /tmp/report.md`.
> The `lookup_only` row is measured against the golden's curated `lookup` lines (the reference grep
> tier), not auto-generated `LOOKUP.md`. The LLM-judge column reflects model grading and can be
> inflated for libraries with strong training-data priors (fastapi/pydantic) — treat it as
> indicative, not absolute.

Generated by `python bench/bench.py`. Token estimate = chars/4 (`scripts.manifest.est_tokens`). **Recall gate**: every golden `must_have` signature must be present — a single miss is a FAIL regardless of token cost. **Judge**: Claude scores whether each golden Q&A is answerable from the artifact alone (model `claude-sonnet-4-6`, via the `claude` CLI / OAuth or `$ANTHROPIC_API_KEY`); `skipped` if no backend is available.

## Recommended default

**Read tier (wire as doc-indexer default): `compact`** — cheapest strategy that passes the recall gate *and* scores ≥95% on the LLM-judge for every benchmarked library (reliably answers usage questions, not just signature presence).

**Grep tier: `lookup_only`** — ~99.9% reduction; passes the recall gate and answers most existence/signature questions in 1–2 tokens, but scores lower on the judge (it lists APIs without full usage). Use it for "does this exist / what's the signature", fall back to the read tier for "how do I use it".
